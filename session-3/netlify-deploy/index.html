ERA V5 · Session 3 assignment

Bharat-40B — Design Brief

A 40B-parameter, Gemma-4-class model that is excellent at coding, agentic work and Indic languages, and is India-first — it should view the world from an Indian perspective because its corpus and annotators do, not because a system prompt tells it to. Working assumptions: dense 40B, d_model = 6144, ~10T-token pre-training budget (≈12× Chinchilla — over-training is standard for serving-class models).

1 · Data

Pre-training mix — 10T tokens

SliceShareWhat & why
English web + books34%Filtered CommonCrawl, books, encyclopedias — the backbone of general knowledge and reasoning.
Code24%Permissively-licensed GitHub plus PRs, issues, commit diffs, CI logs, notebooks. The PR/issue/diff layer — not raw files — is what teaches repo-level reasoning that agents need.

Math + science9%arXiv, OpenWebMath-class corpora, textbooks, proofs — reasoning depth for coding/agentic work rides on this.

Agentic traces3%Terminal sessions, browser interaction traces, tool/API documentation, structured logs — priors for tool use before any RL.
Synthetic8%Textbook-style generation for low-resource Indic, verified translations of reasoning chains, code explanations in Indic. Used only where organic data runs out, never to replace it.

Post-training (SFT) — ≈2M examples

  • 40% code — multi-turn, repo-level, unified-diff output format (agent-compatible from day one).
  • 25% agentic trajectories — only environment-verified successes; deliberately includes failure-then-recovery turns.
  • 20% Indic instructions — natively authored, never translation-only; includes code-mixed Hinglish/Tanglish because that is how India actually types.
  • 10% India-first QA — civics, law, geography, festivals, current affairs, authored in-country.
  • 5% safety — communal-sensitivity, multilingual jailbreaks, Indian PII norms.

RL / alignment

  • RLVR wherever a verifier exists: code (hidden unit tests), math (answer match), agentic (sandboxed SWE/browser/terminal tasks; outcome reward, small per-step cost to discourage flailing).
  • Preference RL (DPO → online RLHF) with a pan-India annotator pool under explicit quotas (language × region × urban/rural) and a written rubric grounded in constitutional values (pluralism, dignity, secularism). India-first alignment is an annotator-recruitment decision, not a slogan.

2 · Cleaning — ordered pipeline

  1. Language ID trained in-house, robust to code-mixing and romanized Indic — off-the-shelf LID silently deletes Hinglish.
  2. Script normalization: NFC only. Preserve ZWJ/ZWNJ (they change meaning in Indic conjuncts — क्ष vs क्‌ष). Never NFKC on code or math (it rewrites symbols).
  3. Dedup — exact SHA + MinHash, at document and paragraph level, across sources.
  4. Quality filtering per language — a small classifier trained per Indic language. An English-perplexity filter is the classic silent killer of Indic data; it must never gate non-English text.
  5. Code-specific: license allowlist, secret/credential scrub, vendored & generated code removal, parse/compile check, near-clone dedup.
  6. PII + toxicity, India-tuned: Aadhaar/PAN/phone patterns; per-language toxicity thresholds audited by humans so dialects aren't over-deleted.
  7. Decontamination: 13-gram overlap against every benchmark in §3 — including their translations.

4 · Tokenizer — fertility targets → vocabulary size

Language focus

Tier 1: English, Hindi, code · Tier 2: Bengali, Tamil, Telugu, Marathi · Tier 3: Gujarati, Kannada, Malayalam, Punjabi, Odia, Urdu (Sanskrit, Assamese best-effort)

Vocab-size arithmetic (the load-bearing decision)

embedding + LM-head cost = 2 · V · d_model, with d_model = 6144:

Vemb params% of 40BVerdict
65,536 (2¹⁶)0.8B2%cheap, but 12 scripts cannot hit the Indic targets above
131,072 (2¹⁷)1.6B4%targets reachable (validated with small weighted-BPE runs) — chosen
262,144 (2¹⁸)3.2B8%<2% extra sequence compression beyond 131k — diminishing returns

Decision: V = 131,072 — GPU-friendly power of two, 4% parameter tax. Budget: ≈46k Latin/English+code merges · ≈42k Indic (weighted by tier) · ≈10k math/science symbols · ≈32k shared punctuation/digits/multilingual residue · 256 byte-fallback (nothing is ever unrepresentable) · ≈600 reserved specials (tool-call/observation markers, FIM, control).

Method: BPE with per-language corpus weights tuned in a measure-fertility → adjust-weights → retrain loop — the same procedure as our Session-2 tokenizer, scaled up. The fertility dashboard in §3 is the acceptance gate.

Trade-offs owned

  • 24% code + 16% Indic squeezes English web to 34% — accepted: code + math carry reasoning, and English transfers via parallel data.
  • Indic fertility targets bound the vocab from below; the 2·V·d embedding tax bounds it from above — 2¹⁷ is the intersection.
  • Synthetic Indic is capped at 8% and always verifier-checked — low-resource languages are where synthetic data most easily poisons style.
